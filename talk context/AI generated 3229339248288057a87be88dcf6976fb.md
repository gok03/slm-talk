### The problem (why this matters)

1. Latency budgets are brutal (UX, agents, RPA, real-time pipelines)
2. Cost compounds (per-token + retries + orchestration + tool calls)
3. Privacy & data boundaries (PII, contracts, healthcare, enterprise)
4. Reliability: LLM calls fail in the real world
    1. JSON breaks
    2. Tool calls drift
    3. Prompt injection / document weirdness
    4. Long-tail inputs dominate

*If those constraints disappeared, we’d all just use frontier models everywhere — but they won’t.*

---

### SLMs: what counts today?

1. “Small” is contextual:
    1. Parameters (e.g., 1B–8B is common)
    2. Memory footprint (can it run on a laptop/CPU?) + kv cache
    3. Throughput / tokens per second
    4. Cost per successful task (not per request)
2. SLM sweet spot: *narrow tasks with tight constraints*
    1. extraction
    2. classification / routing
    3. summarization with structure
    4. agent steps that need speed + determinism

---

### 3 ways to get small (and when each wins)

1. **Quantizing a large model**
    1. Best when you want “general ability” but need lower cost/memory
    2. Tradeoff: memory ↓, quality ↓ (especially reasoning / long context)
2. **Distilling a large model into a smaller one**
    1. Best when your task is repeatable and you can generate training data
    2. Tradeoff: upfront effort ↑, but per-call reliability ↑
3. **Training / choosing a purpose-built small model**
    1. Best for domain workflows (docs, tickets, forms, RPA)
    2. Tradeoff: narrower; needs good evals

### Benchmarks vs reality

1. Leaderboards overfit what’s easy to measure
2. Your eval should reflect your pipeline:
    1. accuracy on your documents
    2. schema adherence (valid JSON rate)
    3. latency distribution (p50/p95)
    4. cost per successful output
    5. fallback rate to a bigger model

### Closing

1. When NOT to use SLMs
    1. high-stakes complex reasoning with no guardrails
    2. tasks requiring deep world knowledge freshness
    3. open-ended creative generation where “taste” matters most
2. A pragmatic future: *cascades*
    1. SLM as default
    2. escalation to bigger model only on hard cases
    3. continuous evals + data flywheel
3. One takeaway: production AI is an optimization problem — SLMs give you more control knobs.
